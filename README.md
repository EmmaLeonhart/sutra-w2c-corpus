# sutra-w2c-corpus

A **weights↔code** training corpus for [**Sutra**](https://github.com/EmmaLeonhart/Sutra)
weight→code decompilation: generated Sutra programs whose behavior is
carried by matrices, paired with those matrices (the "weights") and the
program's substrate input→output behavior. The long-term goal is a model
that maps **weights → code** (recovering the program from its learned
parameters).

This dataset is generated by `experiments/weight_to_code_corpus.py` in the
Sutra repo (where it is pinned as the `corpus/` submodule) and mirrored
here. Every entry is **self-consistency-tested**: recompiling its `source`
with its `weights` reproduces the recorded `io` on the Sutra substrate.

## Files

- `corpus.jsonl` — one programmatic (template) program per line.
- `gemma_corpus.jsonl` — programs generated free-form by `gemma3:12b` and
  filtered to those that compile + run on the substrate.
- `*.csv` — one file-backed weight matrix per matrix (comma-separated
  floats, one row per line), referenced from a program's `source` via
  Sutra's `load_matrix("…csv")`.
- `model/` — the **official baseline weight→code model** trained on this
  corpus (checkpoint + vocab + full substrate eval): exact source match
  **0.811**, substrate IO reproduction **0.826** on the 720 held-out
  programs, 0 compile/run failures. See `model/README.md` for the
  per-structure breakdown and the coefficient wall.

## Schema (one JSONL line = one datapoint)

```json
{
  "id": "linear_K4_gaussian_s0",
  "structure": "linear",           // program family (template entries)
  "K": 4,                          // input/matrix dimension
  "weight_kind": "gaussian",       // gaussian | perm | trained_rotation | trained_perm
  "seed": 0,
  "source": "function vector apply(vector x) { matrix M0 = load_matrix(\"…csv\"); return Tensor.MatrixMul(M0, x); } …",
  "weights": [{"name": "M0", "csv": "…csv", "shape": [4,4], "kind": "gaussian"}],
  "io": [{"input": [...], "output": [...]}, …],   // computed on the substrate
  "runtime_dim": 4,
  "llm_model": "none"              // model-free programs need no embedding LLM
}
```

Gemma entries (`gemma_corpus.jsonl`) share `source` / `K` / `io` but carry
`generator` (e.g. `gemma3:12b`) and `weights: []` (their matrices are
inline literals in the source rather than separate CSVs).

## Contents (current snapshot)

- **7200 programmatic programs** = 15 structures × 6 dims × 4 weight-kinds × 20 seeds.
  - structures: `linear`, `chain2`, `chain3`, `residual`, `diff`, `scaled`,
    `affine`, `sum2`, `bundle2`, `bundle3`, plus five harder families that
    force inference over template-matching — `chain4`, `scaled_res`
    (`a·M@x + x`), `gen_affine` (`a·M@x + b·x`), `scaled_diff` (`a·M@x − b·x`),
    `two_mat_affine` (`a·M0@x + b·M1@x`) (240 each). The coefficient families
    carry per-program discrete coefficients `a`/`b` ∈ {0.5, 1.0, 1.5, 2.0, 3.0}
    rendered as source literals; a coefficient is only recoverable from
    IO + weights, so it cannot be templated.
  - weight kinds: `gaussian`, `perm`, `trained_rotation`, `trained_perm` (900 each;
    the `trained_*` kinds are matrices trained by gradient descent on the
    substrate toward a structured target — orthogonal / permutation — so the
    weights carry structure, not noise).
  - dims K ∈ {4, 6, 8, 10, 12, 16} (600 each).
  - 5760 weight CSVs, 14400 input→output pairs.
- **8 Gemma-generated programs** (free-form, substrate-validated).

Scale further with a one-flag bump (`--seeds` / `--ks`) on the generator.

## Reproduction

From the Sutra repo:

```bash
py experiments/weight_to_code_corpus.py            # programmatic (template) corpus
py experiments/gemma_codegen_corpus.py             # + Gemma free-form (needs ollama gemma3:12b)
py experiments/mirror_corpus_to_hf.py              # mirror to this HF dataset
```

Programmatic generation is deterministic (seeded). Each entry's
(source, weights) reproduces its `io` on the substrate — guarded by
`experiments/test_weight_to_code_corpus.py` and
`experiments/test_gemma_codegen_corpus.py`.
